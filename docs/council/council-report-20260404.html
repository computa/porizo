Story Guidance Algorithm — April 4, 2026 | 5 advisors, 5 peer reviews, 1 chairman

The Question: What is the optimal algorithm for Porizo's story guidance that knows when "enough is enough," never repeats questions, feels warm, caps at 2-3 rounds, and still produces great lyrics? The current 8-slot + 5-block + 5-stage pipeline is completely broken.

Where the Council Agrees

  • The current system is catastrophically overengineered. 8 slots, 5 blocks, 5-stage pipeline — unanimous: burn it down.
  • One LLM call per round, not five. The 5-call pipeline is indefensible on cost, latency, and complexity.
  • Ready after round 1, hard cap at 2-3. Never block creation after name + occasion + one specific detail.
  • Lyrics generation is where the real work happens. Spend prompt budget on generation, not interrogation.
  • Warm framing, not clinical extraction. "Tell me more" not "please provide additional narrative context."

Where the Council Clashes

Zero follow-ups vs. one round of guided deepening.

The Contrarian, First Principles, and Outsider argue: take whatever the user gives, generate immediately. The Executor and Expansionist argue: one optional round transforms output quality dramatically.

Peer reviews broke the tie: hallucinated personal details in a birthday song are catastrophic (can't invent memories), and occasion-specific complexity matters (funeral tribute needs more than "happy birthday bro").

Verdict: One optional round wins. Pure zero-guidance is too risky for emotional accuracy.

Blind Spots the Council Caught

  1. Gift-giver framing — The user describes someone else, for someone else. Prompt should be "Tell me about [recipient]" not "Tell me your story."
  2. Post-creation enrichment is the unlock — After hearing the preview, users have energy to add detail. Pre-creation = homework. Post-creation = excitement.
  3. Cold-start paralysis — Many users freeze at a blank prompt. A warm opener with scaffolding helps.
  4. Occasion-specific readiness — Birthday needs less context than a memorial tribute. Flat threshold is wrong.
  5. Existing codebase migration — Replacing 500 lines of pipeline without breaking the rest of the app.

The Recommended Algorithm

Two-round guidance with immediate creation access and post-preview enrichment.

0
User's first messageSingle LLM call extracts: {name, occasion, memories[], tone, ready}. Ready = name + occasion + 1 specific detail. If ready: show "Create Song" prominently. If not: proceed to Round 1.
1
One warm follow-up (optional)Same LLM generates one "Yes, And" question framed as excitement. Occasion-aware. "Create Song" stays visible and tappable regardless. If user answers: one more extraction. Now unconditionally ready.
2
Hard capMark ready. Gently suggest creating. No round 3 ever exists pre-creation.
+
Post-preview enrichment (the real depth)After hearing the preview: "Want to add anything? A nickname, a memory, something only they'd get?" This is where guided storytelling value actually lives. Feed additions into reroll/refined render.

Implementation Specifics

  • One LLM call per round. Structured JSON output. No pipeline stages.
  • Occasion-aware readiness (tribute occasions = more cautious with invented details)
  • Lyrics generation prompt gets the real engineering investment
  • Kill Labov profiles, gap analysis, semantic blocks, readiness thresholds
  • ~50-80 lines replacing ~500 lines of pipeline

The One Thing to Do First

Write the single LLM prompt that replaces the entire 5-stage pipeline. One prompt. Structured JSON output: {recipient_name, occasion, memories[], tone, ready, follow_up_question}. Test against 10 real/synthetic user messages spanning casual birthday to memorial tribute. Validate: marks ready when it should, asks one warm question when not, never re-asks what user already said. That prompt IS the algorithm. Everything else is plumbing.

Full Advisor Responses

The Contrarian — "Stop building gatekeepers, build accelerators"

The fatal flaw isn't the algorithm — it's building an algorithm at all. Optimizing for narrative completeness when you should optimize for time-to-song. Any mandatory multi-round flow is wrong. Competitors ship from one prompt. Every additional round is where users close the app.

The 5-stage pipeline per turn (5 LLM calls just for a follow-up question) is insane. That's your latency, cost, and brittleness.

Fix: Default to ready after round 1. Offer one optional deepening question. Track what was said in a simple key-value map. No slots, no blocks, no profiles. Let the lyrics model handle narrative gaps. Stop building gatekeepers, build accelerators.

First Principles — "You're solving the wrong problem"

Why interrogate users at all? Your competitors proved one prompt is enough. Strip to first principles: a great song needs a name, an occasion, and one emotionally specific detail.

The answer isn't a better guidance algorithm — it's a better lyrics generation prompt. Take whatever the user gives and have the LLM infer gaps. If you must ask: one round max, framed as excitement not extraction.

Kill gap analysis, kill readiness threshold. Ship songs faster. Max rounds: 1.

The Expansionist — "Story data is the moat"

The broken algorithm is hiding Porizo's biggest strategic asset. Every completed story produces structured emotional memory data no competitor has. The algorithm should accumulate this asset.

4 moves: (1) Single-pass LLM with fact memory, (2) Ready after R1/great after R3 — show Create immediately, (3) "Yes, And" tone baked in, (4) Store stories for fine-tuning.

Max rounds: 3, pipeline stages: 1, readiness slots: 3. The upside of story extraction done right is enormous.

The Outsider — "You're overengineering a greeting card"

8 slots, 5 stages, 2 profiles — architecture to impress engineers, not serve users. Why gatekeep the user's own story? Competitors ship with zero follow-ups.

Generate after first message. One optional follow-up. Never block. If you have a name, you have enough.

The guided storytelling value comes from output quality, not interrogation length. Make the first song so good they want to add detail for version two.

The Executor — "Replace 500 lines with 50. Ship Monday."

Kill the 8-slot gap analysis, 5-stage pipeline, and dual readiness profiles. All of it.

Round 0: User types. Round 1: Single LLM call extracting name/occasion/memories/tone. Ready if name+occasion+1 detail. Else ONE "Yes, And" follow-up. Round 2: Hard cap.

One call per round. Structured JSON output. Drop Labov. Lyrics generation is where prompt budget goes. Replace ~500 lines with ~50.

Peer Review Highlights

Strongest response: The Executor (4/5 reviewers). Concrete, shippable spec with exact structure.

Biggest blind spot: First Principles — ignores cold-start paralysis (users freeze at blank prompt) and that lyrics LLM can't safely invent personal memories.

What ALL missed: (1) Gift-giver framing changes prompts fundamentally. (2) Post-creation enrichment is higher-conversion than pre-creation interrogation. (3) Occasion-specific readiness thresholds. (4) Migration within existing codebase.
