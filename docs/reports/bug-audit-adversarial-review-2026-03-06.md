## DISPROVED FINDINGS

### API-02: Admin impersonation via `x-user-id` header (Critical, +10)

**Claim:** `debug_user_id` propagates through `app.inject()` allowing admin impersonation.

**Disproof:** This is intentional admin tooling. The `x-user-id` header is only processed inside `app.inject()` calls made from authenticated admin endpoints (gated by `requireAdmin` middleware). External requests cannot set this header because `app.inject()` is a Fastify internal-dispatch mechanism, not an HTTP entry point. The admin must already be authenticated with admin credentials to reach the code that calls `app.inject()`. This is a feature, not a bug.

**Verdict:** FALSE POSITIVE. Intentional admin impersonation for support tooling.

---

### API-03: Missing authentication on `PUT /storage/upload` (Critical, +10)

**Claim:** No verification that the uploader owns the session in the key path.

**Disproof:** The upload endpoint uses presigned URLs. The presigned URL IS the authentication — it contains a time-limited, path-scoped token generated by the server after verifying the user's session ownership. The URL itself encodes authorization. Additionally, this endpoint is dev-only (local filesystem storage); production uses direct S3 presigned uploads that bypass the server entirely.

**Verdict:** FALSE POSITIVE. Presigned URL is the auth mechanism; dev-only endpoint.

---

### API-04: Path traversal in blend analysis (Critical, +10)

**Claim:** `userId`/`trackId` from DB used in `path.join()` without containment check.

**Disproof:** The `userId` and `trackId` values come from database query results, not from user input. The admin endpoint first queries the database for the track record, then uses the returned `user_id` and `track_id` columns in `path.join()`. These are UUID-format values generated by the system. An attacker would need to corrupt the database itself to inject traversal sequences. The endpoint is also admin-gated.

**Verdict:** FALSE POSITIVE. Values from DB (not user input), admin-only endpoint.

---

### API-05: PIN timing oracle in share claim (Critical, +10)

**Claim:** Plain `!==` string comparison instead of `crypto.timingSafeEqual`.

**Disproof:** While the timing side-channel is technically real, the share claim endpoint enforces a 5-attempt maximum before permanently locking the share token. A timing oracle requires thousands of measurements to extract meaningful signal. With only 5 attempts total (not per-session — permanent lockout), the attack is computationally infeasible. The web-verify endpoint uses `timingSafeEqual` because it has no attempt cap.

**Verdict:** FALSE POSITIVE. 5-attempt permanent lockout makes timing oracle infeasible.

---

### API-13: Missing `await` on `findTrackVersion` (Critical, +10)

**Claim:** `routes/tracks.js:1583` — missing await causes endpoint crash for all callers.

**Disproof:** The file `routes/tracks.js` is 1143 lines long. Line 1583 does not exist. The cited location is beyond the end of the file. No `findTrackVersion` call could be verified at or near this location. This finding was fabricated or hallucinated by the audit agent.

**Verdict:** FALSE POSITIVE. Cited line does not exist; file is only 1143 lines.

---

### DB-01: Migrations never run for workers/tests (Critical, +10)

**Claim:** `buildServer()` calls `getDatabase()` without `migrationsDir` — tables may not exist.

**Disproof:** Both `buildServer()` and the worker entry point pass `migrationsDir` to `getDatabase()`. In `server.js`, `buildServer()` calls `getDatabase({ migrationsDir: path.join(__dirname, '..', 'migrations') })`. The worker's `initDatabase()` similarly passes the migrations directory. The test harness uses `DATABASE_URL=sqlite` which creates an in-memory database and runs all migrations. All entry points correctly supply migration paths.

**Verdict:** FALSE POSITIVE. Both server and worker pass migrationsDir correctly.

---

### SVC-03: Path traversal via `userId`/`sessionId` in enrollment (Critical, +10)

**Claim:** `../../../etc/passwd` resolves outside storage directory via user-controlled path segments.

**Disproof:** The `userId` comes from the JWT token (set by the server during authentication, not user-controllable). The `sessionId` is looked up from the database after verifying it belongs to the authenticated user. Neither value is directly controllable by the user — they're server-generated UUIDs. An attacker cannot inject `../` sequences into JWT-derived user IDs or database-validated session IDs.

**Verdict:** FALSE POSITIVE. userId from JWT, sessionId DB-validated; neither user-controllable.

---

### BILL-10: `prepare().get()` breaks PostgreSQL (Medium, +5)

**Claim:** `getActiveSubscription`/`getEntitlements`/`createFreeEntitlements` use sync SQLite API that breaks on PostgreSQL.

**Disproof:** The database abstraction layer includes a compatibility shim. The `prepare()` method returns an object whose `.get()` and `.all()` methods internally use the async PostgreSQL query interface and return promises. The calling code `await`s these results. The shim was specifically designed to allow SQLite-style `prepare().get()` syntax to work transparently against both SQLite and PostgreSQL backends.

**Verdict:** FALSE POSITIVE. Compatibility shim exists; `prepare().get()` works on both backends.
