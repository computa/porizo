# Reviewer: data-migrations (migrations 097, 098)

## Findings (11 total)

### HIGH

1. **[HIGH] migrations/pg/097_voice_provider_profiles.sql:1-67 — Missing foreign keys**
   - Issue: `voice_provider_profiles.user_id`, `voice_provider_profiles.voice_profile_id`, `voice_provider_jobs.user_id`, `voice_provider_jobs.voice_profile_id`, and `voice_provider_jobs.voice_provider_profile_id` have NO FK constraints. Orphan rows possible. Violates song-transfer atomicity rule (per `~/.claude/rules/porizo-project_song_transfer_checklist.md`).
   - Fix: Add `REFERENCES users(id) ON DELETE CASCADE` on user_id, `REFERENCES voice_profiles(id) ON DELETE CASCADE` on voice_profile_id in both `voice_provider_profiles` and `voice_provider_jobs`. Add `REFERENCES voice_provider_profiles(id) ON DELETE SET NULL` on `voice_provider_jobs.voice_provider_profile_id`. Mirror in SQLite migration.

2. **[HIGH] migrations/pg/097_voice_provider_profiles.sql:42-43 — voice_provider_jobs.status / step have no CHECK constraint**
   - Issue: `voice_provider_profiles.status` enforces enum via CHECK (lines 7-12), but `voice_provider_jobs.status` (default 'pending') and `step` (default 'prepare_persona') have NO CHECK. Typos write silently. Drift risk.
   - Fix: Add CHECK constraints listing allowed `status` and `step` values to `voice_provider_jobs` in both migrations/097.sql and migrations/pg/097.sql. Keep identical across SQLite/PG.

3. **[HIGH] migrations/098_enrollment_sessions_consent_scopes.sql:13 — SQLite ADD COLUMN not idempotent on replay**
   - Issue: `runMigrations()` runs on every service boot. SQLite has no `ADD COLUMN IF NOT EXISTS`. If ALTER succeeds but UPDATE backfill fails (or process killed mid-migration), `schema_migrations` row is not inserted; next boot re-runs ALTER and fails with `duplicate column name: consent_scopes`. PG side has IF NOT EXISTS so it's safe.
   - Fix: Either (a) verify `runMigrations()` wraps each migration in BEGIN/COMMIT with rollback on failure, OR (b) split into two migrations: `098_add_column.sql` (just ALTER) and `099_backfill_consent_scopes.sql` (just UPDATE).

### MEDIUM

4. **[MEDIUM] migrations/pg/097 vs migrations/097 — Index direction drift**
   - Issue: PG uses `(user_id, provider, status, created_at DESC)`; SQLite uses default ASC. The 098 backfill uses `ORDER BY vpp.created_at DESC` on both — PG index serves it; SQLite scans/sorts. Silent schema drift between test and prod environments.
   - Fix: Add `DESC` to the SQLite index too: `ON voice_provider_profiles (user_id, provider, status, created_at DESC)`. SQLite supports it from 3.3+.

5. **[MEDIUM] migrations/pg/097_voice_provider_profiles.sql:36-55 — Missing index on (status, next_attempt_at) for worker poll hot path**
   - Issue: `voice_provider_jobs` is a worker queue (`attempts`, `max_attempts`, `locked_at`, `locked_by`). Only status-related index is `(status, provider, updated_at)`. Worker fetching `WHERE status='pending' AND (next_attempt_at IS NULL OR next_attempt_at <= now())` won't use it efficiently. Sequential scan on every poll as volume grows.
   - Fix: Add `CREATE INDEX idx_voice_provider_jobs_poll ON voice_provider_jobs (status, next_attempt_at) WHERE locked_at IS NULL;` (PG partial index; SQLite mirror without WHERE).

6. **[MEDIUM] migrations/pg/098_enrollment_sessions_consent_scopes.sql:22-34 — Unbounded backfill UPDATE on boot**
   - Issue: No LIMIT, no batching. On a populated `enrollment_sessions` table the join with `DISTINCT ON` can take row-level locks on every NULL row simultaneously. Will block boot if table grows large.
   - Fix: Either (a) confirm `runMigrations` wraps in transaction AND row count is small (<10k), OR (b) chunk backfill in application code with follow-up script. Document safe-on-prod size assumption in migration header.

7. **[MEDIUM] migrations/pg/098_enrollment_sessions_consent_scopes.sql:15-16 — consent_scopes nullable TEXT with no default and no CHECK**
   - Issue: Application may write `''`, `'[]'`, or `'null'` (string). Three encodings of "no consent" possible. Repeats the loose-typing pattern that caused the silent-deny bug 098 was meant to fix. Regression risk.
   - Fix: Add CHECK that value is NULL or JSON array string: `CHECK (consent_scopes IS NULL OR consent_scopes LIKE '[%]' OR consent_scopes ~ '^[a-z_0-9]+$')` — or commit to JSONB on PG, TEXT-as-JSON on SQLite. Document encoding in column comment.

8. **[MEDIUM] migrations/pg/098_enrollment_sessions_consent_scopes.sql:22-34 — Backfill grants consent retroactively**
   - Issue: For users with multiple `voice_provider_profiles` (different providers, different consent scopes over time), `DISTINCT ON` returns one arbitrary "latest" scope and applies it to ALL the user's NULL `enrollment_sessions` rows — including older ones recorded under a different consent regime. Wrong direction for a fail-secure feature gate.
   - Fix: Either (a) leave existing rows NULL and force re-consent (fail-secure, simpler), OR (b) backfill only `enrollment_sessions` where `session.created_at >= matching voice_provider_profile.created_at`. Pick (a) unless there's UX cost data justifying (b).

### SUGGESTION

10. **[SUGGESTION] migrations/pg/097_voice_provider_profiles.sql:63-67 — voice_provider_locks has no expires_at TTL**

    - Fix: Add `expires_at TIMESTAMPTZ NOT NULL`. Workers `DELETE WHERE expires_at < now()` before acquiring.

## SQLite/PG Sync Diff

**097:**

- SQLite uses TEXT for timestamps; PG uses TIMESTAMPTZ (acceptable per project pattern, worth a header comment)
- Index direction drift: PG `created_at DESC` vs SQLite default ASC
- No other structural drift

**098:**

- PG: `ADD COLUMN IF NOT EXISTS` (idempotent); SQLite: `ADD COLUMN` (NOT idempotent)
- PG: `DISTINCT ON` for backfill subquery; SQLite: correlated subquery + `LIMIT 1`. Equivalent but textually different — drift risk
- Both add column as plain nullable TEXT with no default/CHECK — same loose typing

## Residual Risks

- `runMigrations()` transaction semantics in `src/database/index.js` not verified — SQLite 098 only safe-on-replay if each migration runs in its own tx with rollback
- `enrollment_sessions` table size on Railway unknown — unbounded UPDATE may block boot
- Consent_scopes backfill assumes user-level consent maps to session-level — unsound if product needs per-session consent
- `voice_provider_profiles.metadata_json` and `step_data` are TEXT not JSONB — no validation, no GIN — slow analytics
